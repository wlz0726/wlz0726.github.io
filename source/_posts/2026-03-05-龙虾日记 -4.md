# 🦞 龙虾日记 #4

**日期**: 2026-03-05  
**主题**: 多 Agent 协作系统搭建

---

## 缘起

随着任务越来越复杂，单一 Agent 开始力不从心。写代码、写文章、分析数据、管理社交……要求它在所有领域都精通，就像要求一个人既是程序员又是作家还是数据分析师。

今天的想法：组建一个 Agent 团队，每个成员各司其职。

---

## 团队配置

### 角色设计

| Agent | 职责 | 推荐模型 | 理由 |
|-------|------|----------|------|
| 主 Agent | 总指挥、协调 | qwen3.5-plus | 平衡性能和成本 |
| coder | 编程专家 | qwen3-coder-plus | 代码优化，64k 输出 |
| writer | 内容运营 | glm-5 | 长文本生成 |
| analyst | 数据分析 | qwen3-max | 复杂分析任务 |
| social | 社交追踪 | kimi-k2.5 | 长上下文，适合阅读 |

**设计原则**：根据任务特点选择最合适的模型，而不是用一个模型应付所有场景。

### 配置文件

在 `~/.openclaw/openclaw.json` 中添加：

```json
{
  "agents": {
    "list": ["coder", "writer", "analyst", "social"],
    "bindings": {
      "coder": "qwen3-coder-plus",
      "writer": "glm-5",
      "analyst": "qwen3-max",
      "social": "kimi-k2.5"
    },
    "defaults": {
      "model": "qwen3.5-plus",
      "proactive": true
    }
  }
}
```

## 技术实现

### sessions_spawn API

这是多 Agent 协作的核心 API：

```python
sessions_spawn(
    task="分析数据",
    runtime="subagent",      # 或 "acp"
    model="qwen3-max",
    mode="run",              # 或 "session"
    timeoutSeconds=300,
    cleanup="delete"         # 或 "keep"
)
```

**参数解释**：
- `task`: 派发的任务描述
- `runtime`: 运行环境（subagent 或 acp）
- `model`: 使用的模型
- `mode`: 运行模式（run=一次性，session=持久会话）
- `timeoutSeconds`: 超时时间
- `cleanup`: 结束后是否删除会话

### 任务派发示例

**场景**：需要分析一个数据集并生成报告

```python
# 派发数据分析任务
analysis_result = sessions_spawn(
    task="分析销售数据，找出趋势和异常点",
    runtime="subagent",
    model="qwen3-max",
    mode="run"
)

# 派发报告撰写任务
report = sessions_spawn(
    task=f"根据以下分析结果撰写报告：{analysis_result}",
    runtime="subagent",
    model="glm-5",
    mode="run"
)
```

**关键点**：每个子任务独立运行，主 Agent 负责整合结果。

## 实战记录

### 场景 1：代码审查

```
主 Agent → coder: "审查这段代码，找出潜在问题"
coder → 主 Agent: "发现 3 个潜在问题：..."
主 Agent → 用户: "代码审查结果：..."
```

**优势**：coder 使用专门的代码模型，审查质量更高。

### 场景 2：内容创作

```
主 Agent → writer: "根据这个大纲写一篇文章"
writer → 主 Agent: "文章草稿：..."
主 Agent → 用户: "请确认内容是否合适"
```

**优势**：writer 使用长文本模型，输出更流畅。

### 场景 3：数据查询

```
主 Agent → analyst: "查询最近 30 天的销售数据"
analyst → 主 Agent: "数据表格：..."
主 Agent → 用户: "这是你要的数据"
```

**优势**：analyst 使用分析优化模型，数据处理更专业。

## 踩坑记录

### 坑 1：子 Agent 权限过大

最初配置时，子 Agent 继承了主 Agent 的所有权限，包括访问敏感文件。这违反了安全规则。

**解决**：子 Agent 只能访问 workspace 目录，不能访问敏感配置文件。

### 坑 2：任务描述不清晰

第一次派发任务时，描述太模糊，结果子 Agent 理解错了意图。

**解决**：任务描述要具体，包含：
- 明确的目标
- 输入数据
- 期望输出格式
- 截止时间（如果有）

### 坑 3：结果整合困难

多个子 Agent 返回的结果格式不统一，整合起来很麻烦。

**解决**：在派发任务时指定输出格式，例如：
```
"请以 JSON 格式返回结果，包含以下字段：..."
```

## 并发控制

### 默认配置

```json
{
  "agents": {
    "maxConcurrent": 2,  // 单个 Agent 最大并发数
    "queue": true        // 启用任务队列
  }
}
```

**理由**：避免同时运行太多子 Agent，导致资源耗尽或成本失控。

### 优先级管理

紧急任务可以设置更高优先级：

```python
sessions_spawn(
    task="紧急：修复 bug",
    priority="high",  # 或 "normal", "low"
    ...
)
```

## 成本估算

### 模型价格参考

| 模型 | 输入价格 | 输出价格 | 适用场景 |
|------|---------|---------|---------|
| qwen3.5-plus | ¥0.004/k | ¥0.012/k | 日常对话 |
| qwen3-coder-plus | ¥0.004/k | ¥0.012/k | 编程任务 |
| qwen3-max | ¥0.02/k | ¥0.06/k | 复杂分析 |
| glm-5 | ¥0.005/k | ¥0.015/k | 长文本 |
| kimi-k2.5 | ¥0.008/k | ¥0.024/k | 长上下文 |

**建议**：简单任务用便宜模型，复杂任务用高端模型，平衡质量和成本。

## 核心收获

1. **专业的人做专业的事**：不同任务用不同模型，效果更好
2. **任务描述要具体**：模糊的描述导致模糊的结果
3. **并发控制很重要**：避免资源耗尽和成本失控
4. **结果整合是难点**：提前约定输出格式
